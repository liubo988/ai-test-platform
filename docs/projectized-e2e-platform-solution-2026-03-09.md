# AI E2E 平台项目化升级方案（2026-03-09）

## 1. 结论

本次需求总体可行，且建议直接采用“项目 -> 模块 -> 测试任务 -> 测试计划 -> 执行记录”的分层模型升级现有系统。

原因：
- 现有系统已经具备任务、计划、执行的基本闭环，不需要推翻重做。
- 当前主要问题不在执行能力，而在数据组织方式和产品形态过于扁平。
- 只要把“项目层”和“模块层”补齐，后续权限、统计、协作、Agent 增强都能自然挂接。

## 2. 最优方案

### 2.1 核心分层

推荐实体层级：

1. `Project`
   - 测试项目
   - 承载统一认证、封面、描述、统计
2. `Module`
   - 项目下的业务模块
   - 承载左侧菜单、排序、模块级聚合
3. `TestConfig`
   - 具体测试任务
   - 承载目标 URL、任务描述、任务级计划与执行
4. `TestPlan`
   - 自动生成的测试计划与脚本
5. `Execution`
   - 计划执行记录、过程事件、产物

### 2.2 认证设计

推荐采用“项目级认证 + 任务继承”的方式：

- 登录信息提升到项目级
- 项目下所有任务默认继承
- 补充 `loginDescription` 字段，用来描述登录页 tab、切换顺序、禁用方式等

这样可以解决：
- 同一项目重复维护登录凭证
- 多 tab 登录页自动化策略不明确
- 页面分析和执行阶段登录策略不一致

### 2.3 UI 结构

推荐页面结构：

- 首页：项目卡片墙
- 项目详情页：
  - 左侧模块导航，支持收起
  - 右侧任务卡片列表
  - 比例 3:7
- 执行详情页：
  - 去掉低价值说明
  - 聚焦任务上下文、LLM 对话、实时画面、执行事件、执行产物

## 3. 已识别的必改项

### 3.1 数据库

需要新增：
- `test_projects`
- `test_modules`

需要补 `project_uid` 的表：
- `test_configurations`
- `test_plans`
- `test_plan_cases`
- `test_executions`
- `llm_conversations`
- `execution_stream_events`
- `execution_artifacts`

需要补 `module_uid` 的表：
- `test_configurations`

### 3.2 历史迁移

推荐迁移策略：

- 为历史数据自动创建一个“历史迁移项目”
- 将旧 `module_name` 映射为模块表记录
- 把所有历史任务、计划、执行、会话、事件、产物统一补齐 `project_uid`

这样可以避免老数据不可用或直接报错。

## 4. Agent 化与 Skills 化评估

### 4.1 不建议先 Agent 化的部分

以下部分必须先做成确定性系统，不应该依赖 LLM：

- 项目 CRUD
- 模块 CRUD
- 任务 CRUD
- 计划与执行的主数据归属
- 认证继承规则
- 历史迁移逻辑

原因：
- 这些是平台底座
- 一旦交给 Agent 决策，会带来不可控状态和数据一致性风险

### 4.2 适合 Agent 化的增强能力

以下能力适合做第二阶段增强：

- 自动识别登录页类型
- 根据 `loginDescription` 生成登录步骤建议
- 页面分析后推荐模块划分
- 执行失败归因
- 自动生成修复建议
- 自动补充边界测试点
- 自动生成执行总结

### 4.3 适合 Skills 调用化的能力

可沉淀为独立 skill 的能力：

- 页面分析 skill
- 登录策略识别 skill
- 测试计划生成 skill
- 执行诊断 skill
- 报告总结 skill

推荐方式：
- 平台核心流程仍由确定性接口驱动
- skill 作为可插拔增强服务挂载
- 不让 skill 直接改主数据，只返回建议、分析或补充内容

## 5. 容易遗漏但建议补上的能力

### 5.1 治理类

- 归档与恢复机制
- 审计日志
- 谁创建、谁修改、谁执行
- 敏感信息加密与权限控制

### 5.2 协作类

- 项目成员与角色
- 项目邀请页简化
- 模块负责人
- 任务负责人

### 5.3 统计类

- 项目级通过率
- 模块级失败率
- 最近执行趋势
- 高失败任务排行

### 5.4 资源治理

- 执行事件分页
- 大体积产物清理策略
- 历史脚本存储策略
- 数据保留周期

## 6. 分阶段实施建议

### 阶段一：平台结构升级

- 项目表、模块表、迁移脚本
- 首页项目卡片
- 项目详情页左右布局
- 模块与任务完整 CRUD
- 项目级认证继承
- 计划/执行链路补齐 `project_uid`

### 阶段二：产品化增强

- 项目统计面板
- 更细粒度执行筛选
- 项目级权限与审计
- 邀请页与协作页重构

### 阶段三：智能化增强

- 登录策略建议 Agent
- 执行失败归因 Agent
- 测试覆盖补全 Agent
- 产物总结与报告 Agent

## 7. 最终建议

最优路线不是先把系统做成“重 Agent 平台”，而是先完成项目化主干，再让 Agent 和 Skills 作为增强层接入。

简单说：

- 核心数据层要确定性
- 核心流程要可审计
- Agent 负责增强，不负责主数据真相

这是当前阶段成本、稳定性、可扩展性最平衡的方案。
